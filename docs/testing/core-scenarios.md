# 核心场景测试与发布保障

## 文档职责

本文是系统不变量、测试分层、核心用户旅程和发布判定的权威质量策略。产品验收目标
见[产品需求](../product-requirements.md)，里程碑状态和阶段出口见
[项目计划](../project-plan.md)。

## 目标

测试围绕使用者真实完成康复追踪的完整流程设计，不以页面数量或代码覆盖率代替
产品正确性。任何版本至少要保证：计划显示正确、记录不丢不重、风险反馈及时、
慢网不阻塞交互、离线可以恢复、私人数据不会留在错误的位置。

测试只使用匿名构造数据。公共仓库、测试日志、截图、录屏和 CI 产物不得包含真实
计划、疼痛记录、训练历史、账号或 Token。

## 必须始终成立的系统不变量

这些不变量是发布门禁，不能因某个功能暂未完成而放宽：

1. **按日期读取计划**：历史日期使用当时有效的计划版本；新版本不能改写旧记录，
   生效日前也不能提前出现。
2. **用户确认才算完成**：Garmin、AI、自动匹配和同步任务均不能替使用者勾选
   训练完成。
3. **一次命令一次结果**：同一 idempotency key 重放任意次数，只产生一份事件和
   一个规范结果。
4. **写入原子性**：任务投影、追加事件和 GitHub outbox 要么一起成功，要么一起
   失败，不能出现页面报错但任务已经悄悄改变。
5. **反馈只追加不覆盖**：同一天可以有晨间、训练后、次日和突发等多次反馈。
6. **红灯不依赖网络或 AI**：确定性安全规则在本机和服务端得到相同结果；AI 不能
   把红灯降级。
7. **缓存不是权威数据**：界面区分本机待同步与服务器已确认；迟到响应不能覆盖
   更新的选择或记录。
8. **私人数据隔离**：认证页面和 API 响应不进入 Service Worker Cache；退出登录
   和清除本机数据会移除 Query Cache、IndexedDB 及用户专属状态。
9. **外部故障不级联**：Garmin、DeepSeek、GitHub、Push 或 Cron 失败时，今日计划、
   训练记录和反馈仍可使用。
10. **计划调整可回滚**：建议绑定基础版本；过期建议不可应用；回滚创建新版本，
    不修改历史版本。

## 核心用户旅程

### S01：每天打开并执行训练

**Given** 已登录，今天存在有效计划和若干待完成任务。

**When** 打开主屏 PWA，查看处方，填写实际组次或时长并勾选完成。

**Then** 首屏显示正确日期、计划版本和任务；勾选立即有反馈；服务器确认后状态为
已同步；日历当天完成数随之更新。

重点验证：计划时区日期边界、按日期计划解析、写入原子性、精确缓存更新。

### S02：提交每日及突发反馈

**Given** 当天尚未反馈，或已经有一条早先反馈。

**When** 完成问答并补充主观感受，稍后又提交一次突发反馈。

**Then** 第一次成功后提醒弱化但入口仍存在；两条反馈分别保留；安全级别与确定性
规则一致；红灯提示不等待 DeepSeek。

重点验证：追加语义、表单校验、本机与服务端安全规则一致、敏感日志脱敏。

### S03：查看历史和未来计划

**Given** Tracker 已经存在两个不同生效日期的计划版本。

**When** 在日历快速选择旧日期、新版本生效日和未来日期。

**Then** 日期高亮在 100 ms 内改变；每一天显示当时应生效的版本；旧日期保留原
完成情况；快速点击时最后一次选择获胜，迟到响应不会回写到当前详情。

重点验证：计划时间线、月摘要/日详情分离、请求取消、缓存命中。

### S04：慢网和服务器暂时不可用

**Given** API 延迟两秒或临时返回错误。

**When** 切换底部 Tab、日期、折叠内容或编辑表单。

**Then** 本地交互立即响应；只有相应内容区显示加载或错误；其他区域继续可用；
重试不会制造重复记录。

重点验证：App Shell 保持挂载、TanStack Query 缓存、局部错误边界、幂等。

### S05：离线训练后恢复联网

**Given** 已经缓存近期计划，随后进入飞行模式。

**When** 查看今日计划、勾选任务、提交两次反馈，关闭并重新打开 PWA，再恢复网络。

**Then** 离线记录重启后仍存在并标记为仅本机保存；恢复后按顺序重放；重复触发
启动、online、visible、focus 和手动同步仍只生成一份服务端记录。

重点验证：IndexedDB 持久化、命令队列、idempotency key、重放恢复。

### S06：出差或器械受限

**Given** 某日期范围启用了出差执行上下文，当天时间和器械条件不足。

**When** 选择一个私有备选方案或微训练，并记录实际完成情况。

**Then** 当天显示基础计划和采用的执行方式；基础计划版本没有被改写；第二天可以
选择不同条件；实际记录引用当天上下文。

重点验证：执行上下文覆盖、日期范围、同日优先级、基础计划不可变。

### S07：暂停、返程与继续训练

**Given** 因生病、急性反应或行程暂停若干天。

**When** 结束暂停并查看接续预览。

**Then** 暂停日不会被当作正常未完成；返程不会自动推进阶段；只有确认确实改变
未来任务时才生成新计划版本，历史记录保持不变。

重点验证：暂停/恢复事件、有效训练周、接续差异、版本生效日期。

### S08：AI 提出计划调整

**Given** 反馈和训练记录已经成功保存，当前计划版本为 vN。

**When** DeepSeek 返回结构化建议，使用者查看差异并接受、拒绝或稍后处理。

**Then** 非法 JSON、未知操作和越界修改被拒绝；接受时再次校验基础版本并原子创建
vN+1；拒绝不改计划；等待期间版本变化会让旧建议过期；红灯建议不能自动应用。

重点验证：Provider 故障、Schema、安全规则、并发确认、版本回滚。

### S09：Garmin 同步与任务关联

**Given** 计划开始日后存在活动和基础睡眠记录。

**When** 自动同步、立即同步或重叠两天窗口再次抓取，并尝试关联某个任务。

**Then** provider record ID 相同的记录只保存一次；同步只更新外部数据区域；建议
关联必须人工确认；确认关联也不会自动勾选任务完成。

重点验证：同步游标、分页、幂等 upsert、关联状态、认证失效提醒。

### S10：GitHub 镜像延迟或失败

**Given** PostgreSQL 正常，但 GitHub 返回超时、权限错误或同路径 `409`。

**When** 保存任务或反馈。

**Then** 用户操作仍显示服务器已保存；镜像单独显示待处理或失败；后台串行重试，
最终文件内容与数据库一致且不会重复事件。

重点验证：事务内 outbox、租约/并发领取、同路径串行、重建镜像。

### S11：退出、重新登录和 PWA 更新

**Given** 本机保存了查询缓存和离线记录，并安装了旧版本 PWA。

**When** 退出登录、清除本机数据或安装新版本。

**Then** 未授权页面和 Cache Storage 无法读取私人内容；IndexedDB 和 Query Cache
按身份清理；新版本不会继续展示旧的认证首页；未同步记录存在时清理操作必须先
明确警告。

重点验证：身份变化、缓存 buster、数据库迁移、Service Worker 生命周期。

### S12：阶段与结项评估

**Given** 到达计划目标日期，期间存在暂停和降级周。

**When** 打开应用并填写阶段评估。

**Then** 系统只开启评估，不自动宣布康复完成；有效训练周正确；选择维护、进阶、
延长或人工复评后，只有需要改变未来任务的分支才创建计划版本。

重点验证：目标日期触发、左右侧独立记录、结果分支、历史时间线。

## 自动化测试分层

| 层级                | 主要工具                 | 验证内容                                    | 运行时机       |
| ------------------- | ------------------------ | ------------------------------------------- | -------------- |
| 领域规则测试        | Vitest                   | 日期、红黄绿、计划变更、同步窗口、Schema    | 每次提交       |
| Module 契约测试     | Vitest + 内存 Adapter    | 计划时间线、命令幂等、离线重放、作业状态机  | 每次提交       |
| 交互测试            | Testing Library + Vitest | 日期/Tab 即时反馈、表单、局部加载、迟到响应 | 每次提交       |
| PostgreSQL 集成测试 | 临时 PostgreSQL          | 约束、迁移、原子命令、并发、outbox          | 合并前与主分支 |
| 浏览器旅程测试      | Playwright               | 登录替身、今日、日历、慢网、离线、缓存清理  | 主分支与发布前 |
| 生产冒烟测试        | 最小只读检查             | 健康检查、登录门禁、静态资源、版本信息      | 每次部署后     |
| iPhone 真机清单     | 主屏 PWA                 | 安装、刷新/更新、飞行模式、恢复、输入体验   | 重要版本发布前 |

PostgreSQL 集成测试使用匿名临时数据库和正式 migration，不连接私人 Neon 数据库。
浏览器测试使用匿名计划夹具和测试登录 Adapter，不依赖真实 GitHub OAuth、Garmin、
DeepSeek 或 GitHub 私仓。各外部服务通过可注入 Adapter 模拟成功、超时、限流、
认证失效和非法响应。

## 第一阶段必须自动化的用例

以下用例随 P0 基础切片一起落地，不能留到全部功能完成以后：

| ID    | 用例             | 关键断言                                 |
| ----- | ---------------- | ---------------------------------------- |
| P0-01 | 按日期解析 v1/v2 | 生效日前后和历史日期分别得到正确版本     |
| P0-02 | 任务命令原子写入 | 任务、事件、outbox 全成或全败            |
| P0-03 | 重复命令         | 相同 key 重放只产生一个结果              |
| P0-04 | 日期快速切换     | 100 ms 内高亮；最后选择获胜              |
| P0-05 | Tab 慢网切换     | 页面框架立即出现，网络只影响内容区       |
| P0-06 | 同月缓存         | 切换日期不重新请求月摘要                 |
| P0-07 | 私人缓存规则     | Service Worker 不保存认证首页和 API 响应 |
| P0-08 | 退出清理         | Query、IndexedDB、用户状态均不可再读     |
| P0-09 | 计划时区边界     | 上海 23:59/00:00、UTC 和设备跨区不串日期 |
| P0-10 | 外部故障隔离     | 模拟集成异常不影响今日读取和记录         |
| P0-11 | 单用户身份门禁   | 只按不可变 GitHub 数字 ID 授权           |

P0-10 当前尚未完成。现有实现只证明任务投影、事件与 outbox 能够原子提交，以及核心写入
不直接调用外部服务；它还没有可执行的 GitHub、Garmin、DeepSeek 或响应后消费者边界，
因此不能真实注入“提交后外部尝试失败”。对应消费者落地时，必须在实际编排边界注入异常，
验证数据库事务已经提交、用户回执不被改写、outbox 保持可重试，并且外部错误只更新
集成状态。在此之前，测试名称和完成统计不得把 P0-10 算作通过。

其中 100 ms 只衡量点击到本地选中状态或稳定页面框架，不把远端数据下载时间伪装
成界面性能。自动测试使用延迟 Promise 和受控时钟；发布前再用浏览器慢网和真机
验证，避免只在快速开发机上通过。

## 后续阶段门禁

### P1：当前康复闭环

- S01、S02、S06、S07 全部通过。
- 出差上下文不制造计划版本；暂停后不会自动推进阶段。
- 计划外训练和多次反馈可独立保存、查询和同步。

### P2：离线可靠性

- S05、S11 全部通过。
- 飞行模式、重启、重复重放、冲突和清理均有自动化覆盖。
- iPhone 主屏 PWA 完成一次真实飞行模式恢复验证。

### P3：外部数据与镜像

- S09、S10 全部通过。
- Garmin 重叠窗口无重复；GitHub 故障不阻塞核心闭环。
- 数据库快照可重新生成私有仓库镜像。

### P4：AI 与评估

- S08、S12 全部通过。
- 非法、过期、红灯和并发建议均不能改写计划。
- 接受、拒绝和回滚在计划时间线上可审计。

## 发布判定

一次发布必须同时满足：

1. 静态检查、类型检查、领域规则、Module 契约和交互测试全部通过。
2. 涉及数据库的变更已用正式 migration 完成临时 PostgreSQL 集成测试。
3. 涉及核心旅程的变更完成对应浏览器测试，不允许只靠单元测试。
4. 涉及 PWA、离线、输入或缓存的变更完成 iPhone 真机清单。
5. 生产冒烟测试通过，且没有使用真实健康数据作为自动化测试夹具。
6. 任何已知失败如果会造成数据丢失、重复、越权、错误计划或红灯失效，都禁止
   发布；视觉小问题可以记录后继续迭代。

代码覆盖率用于发现未执行区域，不设置一个脱离风险的统一百分比。关键不变量、
核心用户旅程和故障路径必须有明确断言，不能用“总体覆盖率很高”替代。
