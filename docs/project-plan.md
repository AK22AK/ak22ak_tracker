## 里程碑

### P0：交互与数据基础

目标：界面交互不再依赖每次服务器往返，并建立后续离线和多页面能力的稳定底座。

P0 按三个可独立上线和验证的切片实施，不把正确性、性能和视觉改造塞进一次发布：

#### P0a：安全与数据正确性

状态：当前实现边界已完成并通过数据库集成验证；P0-10 已随首个训记只读 Provider
边界完成故障隔离验证。

- 移除 Service Worker 对认证首页的缓存，并验证退出后无法读取私人内容。
- 按目标日期和固定计划时区解析有效计划版本。
- 客户端生成稳定命令 ID；任务、事件和 outbox 使用 Neon HTTP 非交互事务原子写入。
- GitHub OAuth 白名单从可变 login 迁移为不可变数字用户 ID。
- 生产 `0003` migration journal 的 `created_at/hash` 已与本地 Drizzle migration 核对；
  后续使用只读 `db:journal:check` 作为发布证据。
- 已交付 P0-01、P0-02、P0-03、P0-07、P0-08、P0-09 和 P0-11。
- P0-10 已验证：已提交核心命令后，后续独立的训记同步请求即使遇到 Adapter 限流，
  也只更新集成状态，不回滚或污染任务、事件和 outbox。该结论不包含同一请求内的
  响应后消费者；后续 Provider 仍需各自补故障测试。

#### P0b：导航与客户端数据

状态：原实现已完成正确性验收，但 2026-07-23 的真实线上试用再次发现顶层 Tab 仍受
动态 RSC 往返和全页过渡影响，因此重开为 P0b-r1 性能返修。P0b-r1a 已完成生产验证并
由项目经理验收。使用者确认已访问 Tab 切换已无等待，首次进入仍偏慢但当前不阻塞
使用；P0b-r1b 只保留不扩张范围的最小查询波次补丁，更多性能改造降为观察项。正式
migration、私人策略和聚合 API 结论保持有效；不能再用“100 ms 内显示加载框”代替
缓存内容可见。

- 建立共享受保护 App Shell，四个顶层 Tab 保持挂载和状态。
- 接入 TanStack Query、稳定 Query Key、聚合 API 和精确缓存失效。
- 日期和 Tab 立即响应，详情区独立加载，迟到响应不能覆盖最新选择。
- `TrackerSafetyPolicy` 已迁移到不可变 PostgreSQL 版本；服务端按发生时间解析，聚合
  DTO 鉴权下发，事件记录实际使用的版本/hash。
- 交付 P0-04、P0-05 和 P0-06。

#### P0b-r1a：客户端导航与部署区域返修

状态：已于 2026-07-24 由项目经理完成登录态生产复核并验收。

- 生产基线：今日首次进入日历约 3.1 秒；已访问 Tab 的返回仍约 0.34–1.59 秒。
- 根 Tab 使用受保护 Layout 中的持久客户端 Host；访问过的页面由 React Activity 隐藏
  而不卸载，表单草稿、展开状态和各 Tab 滚动位置继续保留。
- 根 Tab 使用 Next.js 支持的 History API 同步 URL，不再通过动态 RSC 请求完成每次
  切换；直接 URL、服务端鉴权、反馈拦截路由和非根子页面仍保留原路由语义。
- 设置页改为客户端 Query 边界，页面壳不等待集成或镜像状态；今日数据成功后在浏览器
  空闲时精确预取当月、当天和设置状态。
- Vercel Functions 单一区域从默认 `iad1` 调整为 `sin1`，与 Neon 新加坡区域对齐；
  必须以生产构建配置和响应证据确认生效。
- 新门禁要求：未缓存 Tab 在 100 ms 内显示稳定目标壳；缓存 Tab 在 100 ms 内显示真实
  内容，不显示全页 loading，也不重新请求聚合 API。发布后在同一生产环境至少测量五次
  冷/暖路径并报告中位数和最差值。
- 若首次数据可见中位数仍高于 1.5 秒，r1a 不宣称性能收口；下一步只进入 P0b-r1b 的
  PostgreSQL 聚合查询波次优化。
- 正式域名验证覆盖：日历日期查询参数、日历与设置往返、刷新、今日与趋势往返以及
  浏览器前进/后退；各步骤的 URL、可见内容和 `aria-current` 均保持一致，日历选择与
  详情状态能够恢复，登录后控制台无 warning/error。
- 首次稳定壳与已访问页面返回的断裂体验已得到结构性修复。使用者后续确认 Tab 切换
  已无等待，并同意把首次进入速度降为观察项；因此 r1b 只做最小查询波次收口。

#### P0b-r1b：首次数据查询波次最小收口

状态：最小补丁已于 2026-07-24 由项目经理验收；不继续扩大性能范围，不把加载文案
或动画当作数据性能改善。

- 今日聚合在一次查询中取得 Tracker 与目标日期的有效计划版本，减少重复的前置
  PostgreSQL 往返。
- 取得必要上下文后，并行读取当天任务/反馈、安全策略与执行状态；执行状态中的独立
  上下文读取也并行启动。
- 日历聚合在计划版本确定后，并行读取任务、反馈与暂停状态，减少无依赖的串行波次。
- 新增查询编排测试，锁定无依赖读取同时启动；没有修改数据库 Schema、身份门禁、
  计划时间线、安全策略或离线边界。
- 首次真实数据可见速度继续作为体验观察项。只有再次成为使用阻塞时，才重新按鉴权、
  网络、函数、数据库和客户端分段测量，不预先扩张到更多查询重构。

#### P0c：核心界面落地

状态：P0c-1“设计系统基础与今日页任务卡纵向切片”、P0c-2“身体反馈全屏流程”和
P0c-3“日历视觉纵向切片”均已由项目经理验收，P0 核心实现完成。iPhone 长期体验
继续作为持续验收，不阻塞进入 P1。

- 在训记只读纵向切片确认实际训练信息结构后实施，避免把重复手工录入固化为正式
  任务卡。
- 落实已确认的 UI/UX Token、任务卡、反馈卡、日历和状态语言。
- 身体反馈使用稳定 URL 和独立全屏流程；提交成功停留在结果状态，失败保留草稿和
  幂等命令，软导航返回时保留今日任务草稿与滚动上下文。
- 对今日、日历和登录/退出路径完成手机尺寸浏览器测试与 iPhone 真机验收。

#### P0c-r1：面向使用者的文案与信息降噪

状态：已于 2026-07-24 完成生产复核并由项目经理验收。

- 今日、任务展开、身体反馈、日历、设置、待同步、临时训练安排、暂停、接续、离线
  公共壳、趋势和错误/空状态统一改为面向行动的用户语言。
- 默认页面只保留会影响下一步的训练、安全、未保存、离线和待处理事实；架构保证、
  验收结论与开发术语从正常页面移除，异常说明仍给出具体处理动作。
- 训记连接明确使用“训记 API Key”，GitHub 区域使用“数据备份”，本机区域说明实际
  保存内容；没有改变凭证、安全判定、同步状态或数据边界。
- 自动化门禁同时验证正常界面不出现内部术语，并保留疼痛量表、红灯处置、未保存、
  仅本机、清除本机数据等必要说明。

退出条件：当前具备实现边界的 P0 用例全部通过；慢网下日期和 Tab 在 100 ms 内显示
本地选中状态或稳定页面框架。P0-10 已在首个真实外部消费者边界通过；新增 Provider
仍需各自满足相同故障隔离门禁。

### P1：当前康复闭环

目标：覆盖日常执行、出差、暂停和恢复训练的真实使用过程。

#### P1a：训记只读训练输入

状态：P1a-1“只读连接与单日同步底座”、P1a-2“单日训练明细与人工关联闭环”和
P1a-3“计划开始日至今天的追赶同步”均已由项目经理验收，P1a 实现阶段完成。真实
训记 Key 和真实私人数据试用留待使用者以后自行轮换 Key 后进行，不是 P0c 的发布
前置条件。

- 设置页录入轮换后的训记 API Key；服务端验证、认证加密保存并且不回传原值。
- 只调用训练读取接口，按日期同步计划开始日以来的力量训练明细；不接饮食、身体
  数据、官方计划或写回。
- 追赶范围只由服务端依据 Tracker 开始日期、计划时区今天、逐日同步状态和持久游标
  计算；每个请求只处理有界日期批次，设置页在打开期间继续调用，中断后从数据库
  状态恢复。客户端不提交起始日期，失败日期不回滚已成功日期。
- 按稳定来源 ID 和内容 hash 幂等 upsert，重复同步不产生重复记录；来源变化进入
  待复核。
- 今日任务展示训记明细和关联建议，使用者确认、修改或取消关联；任务完成选框保持
  独立，手工训练输入退为兜底。
- 为后续 Garmin 融合保留多来源关联模型，但本切片不同时实现 Garmin 运行时接入。

退出条件：S13 通过；能够从设置页连接、同步一天的匿名训练夹具并确认关联；Provider
认证、限流或超时不影响今日计划、反馈和手工兜底。生产验证只做只读同步，不用真实
训练内容作为测试产物。

- 计划内与计划外训练使用统一实际记录模型。
- 每日、训练后、次日和突发反馈可独立追加。
- 出差/器械受限执行上下文、当天条件与私有备选方案。
- 暂停、返程接续和未来任务差异预览。
- 当前私人计划的新版本导入与历史日期验证。

退出条件：测试场景 S01、S02、S06、S07 通过；临时执行条件不制造计划版本，真正
改变未来安排时才由使用者确认新版本。

#### P1b：临时执行上下文

状态：P1b-1“出差执行上下文与当天条件纵向切片”、P1b-2“暂停状态与暂停日语义”
和 P1b-3“返程/暂停后的接续预览与人工确认”均已由项目经理验收，P1 当前实现阶段
收口。

- 创建或结束出差、器械受限日期范围；计划日始终按 Tracker 的固定计划时区解释。
- 今日页同时展示基础计划与当前上下文，当天条件和私人备选方案可以逐日独立选择。
- 上下文、当天决定、事件和 outbox 原子提交且幂等，不改变计划版本或任务完成状态。
- 私人备选处方只存 PostgreSQL 和私有数据仓库；公共层只包含中性 Schema、导入器、
  鉴权投影和匿名测试。
- 红灯、生病和急性症状不进入普通降级方案，统一显示停止和重新评估提示。
- 暂停作为高于出差/器械受限的独立覆盖层；开始与结束均通过稳定命令、事件和 outbox
  原子审计。暂停日保留基础计划和任务状态，结束后只进入待接续评估。

P1b-1 退出条件：S06 的范围、逐日选择、计划不可变、原子审计、移动布局和安全分流
通过自动化与生产只读验收。后续 P1b 切片再处理暂停、返程和接续差异。

P1b-2 退出条件：S07 前半段的暂停开始/结束、同日执行上下文优先级、计划和任务
不可变、暂停日历语义、原子审计及故障重试通过自动化与数据库集成验证。结束暂停仅
进入待接续评估，返程和接续差异留待后续切片。

P1b-3 交付状态：结束执行上下文或暂停会原子创建待接续评估；今日页和稳定 URL
展示中断摘要、最后确认训练与未来日期差异。按原计划继续只记录决定；顺延只复制
生效日起尚未执行的未来任务到新的不可变计划版本。基线计划变化会让旧评估过期并
基于最新版本重建；评估同时冻结创建时的计划时间线头，任何后来新增的计划版本都会
使旧评估过期。创建评估时已有后续版本则保留“按原计划继续”，禁用不安全的顺延。
所有决定使用稳定命令、事件、唯一决策锁和 outbox 原子审计。正式 `0009`/`0010`
migration、临时 Neon 分支和 S07 后半段集成验证已完成并通过验收。

### P2：离线可靠性

目标：在 iPhone 主屏模式和不稳定网络下不丢计划、训练或反馈。

#### P2a：私人离线只读快照与身份隔离

状态：已由项目经理验收。已交付今日、日历月/日详情及最小安全策略引用的白名单
只读快照；任务和反馈的离线写入由后续 P2b-1 在此基础上扩展。

- Dexie v2 以不可变 GitHub 数字用户 ID、Tracker、查询种类和日期范围隔离快照；
  过期、损坏和其他身份的数据不会恢复。
- 在线鉴权查询成功后保存经过 Schema 白名单投影的数据；完整安全规则、集成凭证、
  Provider raw payload、Cookie 和 Authorization 不进入 IndexedDB。
- 已加载的应用由 TanStack Query 恢复快照；销毁旧 document 后，固定公共离线壳可在
  无 session/RSC/动态 chunk 时读取相同快照，显示今日、月摘要和单日详情的缓存时间与
  只读状态。P2a 验收时所有写操作均要求联网；P2b-1 只放开任务和反馈两类核心命令。
- 退出、身份变化和设置页“清除本机数据”共同清理 Query Cache 与私人 IndexedDB；
  Service Worker 只缓存固定公共离线壳和公共静态资源；认证页面、RSC 和 API 不进入
  Cache Storage。Chromium browser-context 已覆盖断网、关闭页面、从 `/` 冷启动和
  日历浏览；iPhone 主屏 PWA 真机飞行模式仍待验收。
- 公共壳仅在浏览器确实离线时读取快照；在线直达会返回受保护路径重新鉴权。独立最小
  显示契约按 kind 校验身份、Tracker、日期与所有展示字段，损坏的今日、月或单日
  快照分别进入安全空态，不会显示成零任务或零反馈。

#### P2b-1：核心离线命令队列与可靠重放

状态：已由项目经理验收；不据此提前声明 P2b 或 S05 全部完成。

- Dexie v3 将 `pendingCommands` 启用为版本化本机 outbox；只接受任务状态/实际训练
  更新和身体反馈追加，按数字身份与 Tracker 隔离，不保存凭证或 Provider raw payload。
- 在线和离线提交都先落盘，再用确定性投影叠加到今日、单日、月历和公共离线壳；基础
  快照保持服务端权威。反馈只追加，任务连续编辑按创建顺序重放。

退出条件：S09、S10 通过；Garmin 重叠窗口无重复，GitHub 故障不影响核心写入，
PostgreSQL 可以重新生成私有仓库镜像。

### P4：AI、趋势与评估

目标：形成可解释、可确认、可审计和可回滚的计划调整闭环。

#### P4a-1：周完成与症状趋势

状态：已由项目经理验收。

- 服务端按 Tracker 计划时区固定生成包含当前周的最近八个周一至周日自然周，客户端
  不能扩大时间范围；响应明确给出范围、时区和生成时间。
- 任务统计直接读取各日期任务实例，并按当天生效的计划版本过滤；只以任务实例自身的
  `planned/completed/skipped` 状态计数，训记或 Garmin 关联不会被当成完成。
- 每个日期从已保存的追加式身体反馈中取左右膝最高疼痛和最严重安全级别；周汇总同时
  展示反馈天数与应反馈天数，缺失反馈不按零分处理，也不重新推断安全级别。
- 趋势 API 只返回周界限、任务计数、完成率、疼痛、反馈覆盖和绿黄红天数；不返回备注、
  私人阈值、训练处方、外部来源标识或原始数据。
- 趋势页使用轻量 HTML/CSS 图形和同步文字摘要，覆盖首次加载、错误、空数据、部分数据
  与后台更新失败；缓存内容、根 Tab 实例、滚动和其他页面草稿保持不变。
- 本切片不计算训练负荷，不调用 AI，也不包含医学诊断、因果结论、睡眠、阶段评估或通知。

#### P4a-2：训练负荷与数据覆盖趋势

状态：已由项目经理验收。

- 仍按 Tracker 计划时区和当天有效计划版本统计最近八个自然周；负荷只来自用户明确
  完成的任务，待完成、跳过以及未确认的外部记录均不参与。
- 每个任务实例只计一次：时长依次采用手工实际记录、已确认 Garmin 活动、已确认训记
  训练；距离依次采用手工实际记录和已确认 Garmin 活动。来源版本过期、待复核、跨日期
  和与计划无关的记录均排除。
- 同一来源的多条不同记录按稳定记录标识去重后汇总；Garmin 与训记同时关联到同一任务
  时不相加，避免把同一次训练重复计算。
- 周汇总展示完成训练天数、已测量时长与覆盖任务数、已测量距离与覆盖任务数，以及手工、
  Garmin、训记和未测量的任务覆盖。缺少数据保持“未测量”，不解释成零或下降。
- API 只返回周级聚合数字和来源覆盖计数，不返回任务备注、动作明细、外部记录标识、原始
  数据、处方或私人安全阈值；趋势页明确说明训练和身体反馈只是同期展示，不表达因果关系。
- 查询复用 P4a-1 的同一聚合 API，并与任务、反馈和计划版本读取并行；没有新增数据库
  Schema 或 migration，根 Tab、缓存、后台错误保留和草稿不变量保持不变。

#### P4b-1：AI 分析任务与只读计划调整建议

状态：已由项目经理验收；本切片不提供接受、拒绝、应用或回滚。

- 只在使用者明确点击后创建持久分析任务；任务先落库，再在同一请求内进行一次有
  超时和输出大小上限的 Provider 调用。离开页面不保证后台完成，失败和未完成状态
  可以在重新进入后继续查看或重试。
- `PlanAdvisor` 保持 Provider-neutral，DeepSeek Adapter 只在服务端读取环境配置，
  使用 JSON Output，并对空内容、截断、非法 JSON、未知字段、越界操作和安全冲突
  分别失败关闭。模型名由部署配置提供，不写入领域代码。
- 第三方上下文只包含当前有效计划的最小投影、最近十四天的结构化身体反馈和用户
  确认训练；移除身份、仓库来源、反馈备注、训练摘要、外部原始数据和凭证。数据库
  只保存上下文版本、hash、日期范围和模型/响应 hash 等审计信息。
- 每个建议同时绑定基础计划版本和创建时的计划时间线头；任何后来新增的计划版本都会
  使旧建议过期；读取时也会重建当前上下文，新增反馈、红灯、用户确认训练或日期窗口
  变化都会使旧建议失效。确定性红灯优先于模型输出，红灯只显示停止并重新评估，
  不保留训练加码操作。
- 页面提供稳定的只读入口，显示运行、失败、过期、红灯和逐项差异；没有任何计划
  写入按钮。未配置 Provider 时安全显示暂不可用，不发出外部请求。

#### P4b-2a：人工接受/拒绝与不可变计划版本应用

状态：待实施与复核；本切片不提供计划回滚。

- 使用者只能对当前且未过期的建议明确接受或拒绝；接受前由服务端重新核对计划、
  近期记录和安全状态，页面缓存不能替代最终校验。
- 接受时由应用根据数据库中的基础计划与已校验建议生成下一份完整不可变计划版本，
  固定从下一个计划日生效，不修改当天、历史记录或旧计划版本。
- 拒绝只记录决定，不创建计划版本；接受和拒绝都使用稳定命令、原子事件与私有镜像
  outbox，并由数据库唯一约束裁决并发决定。
- 页面先展示差异和生效日，再由使用者二次确认；本切片完成后仍不提供回滚。

#### P4 后续范围

- 建议差异、过期处理、接受/拒绝、计划版本应用和回滚。
- 症状、完成、负荷、睡眠背景和计划版本趋势。
- 阶段/结项评估、有效训练周和下一阶段分流。

退出条件：S08、S12 通过；非法、红灯、过期和并发建议均不能改写计划，接受、拒绝
与回滚在计划时间线上可审计。

## 依赖与风险

| 风险或依赖                      | 处理方式                                                     |
| ------------------------------- | ------------------------------------------------------------ |
| 当前动态路由造成日期和 Tab 延迟 | P0 先完成客户端 App Shell、查询缓存和局部加载                |
| iOS Background Sync 不可靠      | 依靠 IndexedDB，并在多个前台时机重放和显示真实同步状态       |
| Vercel 与 Neon 可能跨区         | 使用生产追踪确认区域，迁移前比较数据库时延和函数配置         |
| Garmin 非官方登录可能变化       | Adapter 隔离、固定版本、Token 失效提示和 FIT/CSV 导入        |
| 训记 Key 权限宽于第一版用途     | 后端加密保存、只读 Adapter、接口白名单和不记录 Authorization |
| DeepSeek 超时、限流或输出异常   | 持久化分析作业、严格 Schema 校验、失败可重试且不影响原始记录 |
| GitHub Contents API 冲突或限流  | outbox 串行目标路径、SHA 重读、退避重试和全量镜像重建        |
| Vercel Hobby Cron 每日且不重试  | 响应后尽力消费、应用/手动触发、持久 outbox 和每日修复        |
| 私人数据泄漏                    | 公私仓分离、服务端 Secrets、最小权限、匿名测试和本机缓存清理 |

风险的架构处理写入对应专题文档；本文只维护其对优先级、交付顺序和发布门槛的影响。

## 发布门禁

每个里程碑进入可用状态前必须满足：

1. 静态检查、类型检查、领域规则和对应自动化测试通过。
2. 数据库变更使用正式 migration，并验证事务、幂等和 outbox。
3. 核心用户旅程完成浏览器测试；PWA、离线、缓存或输入变更完成 iPhone 真机检查。
4. 生产冒烟验证身份门禁、健康检查、静态资源和受影响功能。
5. 不存在会造成数据丢失、重复、越权、错误计划或红灯规则失效的已知问题。

详细用例和分层见[核心场景测试与发布保障](testing/core-scenarios.md)。
