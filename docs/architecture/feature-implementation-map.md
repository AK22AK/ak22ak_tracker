# 功能实现总图

## 目标

AK Tracker 的核心不是“日历加几个表单”，而是一个可离线、可审计、能接入外部
数据和受约束 AI 的单用户追踪系统。所有模块共享相同的数据闭环：

```text
计划 -> 当日任务 -> 人工确认的实际执行 -> 身体/主观反馈
  -> 确定性规则 -> 可选 AI 建议 -> 人工确认 -> 新计划版本
```

膝关节康复是第一个模块；公开代码只保存通用机制和匿名 Schema，私人处方、阈值、
医嘱和实际记录继续留在私有数据域。

## 产品信息架构

### 今日

今日页是默认入口，依次显示：

1. 当前日期、计划版本、执行模式、网络和同步摘要。
2. 必须立即注意的安全或中断提示。
3. 当日任务与实际完成记录。
4. 当日反馈提醒和追加反馈入口。
5. Garmin 待确认关联、AI 待确认建议等次要事项。

### 日历

- 月视图只显示计划数、完成数、反馈标记和异常状态。
- 选择日期后在同页查看计划、实际训练、反馈和数据来源。
- 日期选择是本地交互；月摘要和日详情分别缓存。
- 历史记录可读，未来日期只显示当时有效的计划，不允许误记为已完成。

### 趋势

- 周完成率和有效训练周。
- 左右侧疼痛、肿胀和功能反应趋势。
- 跑步/步行负荷、力量任务完成情况和睡眠背景。
- 计划版本、出差/暂停和黄红灯事件时间线。
- 只展示相关性和时间关系，不宣称因果或给出医疗诊断。

### 设置

- Tracker 与当前计划版本。
- Garmin、DeepSeek、GitHub 镜像和通知状态。
- 离线记录、待同步数量、立即同步和错误处理。
- 计划版本历史、回滚、数据导出、清除本机数据和退出登录。

## 功能矩阵

| 能力          | 界面与交互                                       | 领域/后端实现                                         | 离线与失败边界                                         |
| ------------- | ------------------------------------------------ | ----------------------------------------------------- | ------------------------------------------------------ |
| 单用户登录    | GitHub 登录；未授权只显示登录页                  | 服务端 Session + GitHub login 白名单；每个 API 重验   | 已缓存私人数据仍受本机登录状态控制；退出清空 IndexedDB |
| 今日计划      | 任务待办、处方详情、版本和状态                   | 当前有效计划版本 + 按日任务实例                       | 读取最近缓存；过期时明确标注，不因集成失败不可用       |
| 实际训练      | 计划内逐项记录，也允许计划外训练                 | 结构化实际值 + 主观记录 + 追加式事件                  | 本地 UUID/idempotency key；显示待同步、失败、已同步    |
| 身体反馈      | 每日至少一次提醒，完成后弱化；允许追加和突发反馈 | 模块化表单 Schema + 确定性安全分级                    | 离线照常记录；安全规则在本地可执行，AI 不参与基础判灯  |
| 日历历史      | 月摘要、未来计划、历史执行与反馈                 | 月聚合 DTO 与按日详情 DTO 分离                        | 月和日分别缓存；切换日期不依赖网络                     |
| 提醒          | 页内提醒优先；以后可选主屏推送和角标             | 每日要求状态 + Push subscription + 定时发送           | 拒绝通知不影响功能；推送不是唯一提醒来源               |
| 出差/突发模式 | 选择日期范围和当天可执行条件；显示维持目标       | `execution_context` 覆盖层引用私有规则选项            | 条件每天可变；不因选择一个选项就重写整个计划           |
| 暂停与返程    | 暂停原因、恢复检查、接续方案预览                 | 暂停事件 + 接续决策；真正改变未来任务时生成新计划版本 | AI 可建议，确定性规则和人工确认优先                    |
| 阶段/结项评估 | 评估清单、左右侧分别记录、选择下一阶段目标       | `evaluation_session` + 结果分支 + 新计划或维护状态    | 日历日期不是自动结束条件；评估未完成时保持当前阶段     |
| 计划版本      | 差异、来源、接受/拒绝、回滚                      | 不可变 `plan_versions`；建议绑定基础版本              | 过期建议不能应用；回滚同样创建新版本而非修改历史       |
| Garmin 同步   | 最近成功、立即同步、重新授权、导入文件           | 隔离 `GarminClient` + 游标 + 幂等 upsert              | 自动同步、非官方接口和导入均可替换；失败不影响核心闭环 |
| Garmin 关联   | 活动建议对应任务；用户确认/修改/取消             | `external_record_links` 保存建议与人工决定            | 外部记录永不自动完成任务                               |
| DeepSeek 建议 | 显示原因、影响日期、任务差异和安全级别           | 最小上下文 -> 结构化输出 -> Zod/规则校验 -> 建议记录  | 超时、余额、限流和格式错误均可重试；不丢原始反馈       |
| GitHub 镜像   | 设置页显示正常、延迟或失败                       | PostgreSQL 事务写 outbox；Contents API 串行写目标文件 | 最终一致；镜像失败不把用户操作标记为失败               |
| 趋势分析      | 图表、周摘要、事件时间线                         | PostgreSQL 聚合或物化快照；原始事件保持不变           | 使用缓存的最近结果；不从图表反向改写原始记录           |
| 数据导出/恢复 | 下载结构化快照、查看镜像状态                     | 从 PostgreSQL 生成版本化 JSON；GitHub 作为可读镜像    | 恢复演练验证幂等、引用完整性和 Schema 版本             |
| 多 Tracker    | 设置中选择项目；各模块拥有自己的今日页配置       | 通用核心 + module registry + 模块专用 Schema/组件     | 不为了未知模块把康复字段强行抽象成无类型 JSON          |

## 出差、突发和结项的建模

### 执行上下文覆盖层

出差、缺少器械、只有零碎时间或临时暂停描述的是“今天在什么条件下执行”，并不
等同于基础计划被永久修改。应用保存一个有日期范围的 `execution_context`：

```text
normal | contingency_option | walking_only | micro_session | paused
```

具体可选方案、动作和阈值来自私有计划版本。公开核心只保存选择的稳定标识、范围、
原因和实际执行结果。每天可以选不同条件，不为每次选择制造计划版本。

返程或中断结束后，软件根据最后完整有效阶段、实际完成和安全反馈生成接续预览。
若未来任务需要移动、降低或重复阶段，用户确认后才生成新的不可变计划版本。

### 结项是评估节点

计划的目标日期只触发评估流程，不自动把 Tracker 标为完成。评估至少能记录目标
完成度、症状/功能、力量/控制、负荷耐受和下一阶段意向。评估结果可以进入维护、
专项进阶、延长当前阶段或建议人工复评等分支；所有具体阈值保存在私有规则中。

## AI 调整不是自由 Agent

AI 层实现为受约束的 Proposal Generator，而不是拥有工具权限的自治 Agent：

1. 软件汇总最小化上下文并先执行确定性安全规则。
2. Provider Adapter 调用配置化模型，要求 JSON 或 Tool Call 形式的受限操作。
3. 返回值经过 JSON 解析、Zod、业务规则、基础版本和日期范围校验。
4. 页面展示 diff；使用者接受后由应用自己的服务执行。
5. 保存输入摘要、模型标识、原始输出哈希、校验、决定和结果。

模型名、Base URL 和超时均为服务端配置。DeepSeek 的兼容模型和名称正在变化，
不能把某个模型字符串写进领域逻辑。

## Garmin 接入层级

Garmin 不能成为第一版核心闭环的前置条件。适配器按以下顺序保持可替换：

1. 获批后优先使用 Garmin Connect Developer Program 的 Activity/Health API。
2. 私人使用阶段可试验维护中的非官方客户端，但固定版本、隔离部署、加密 Token，
   并把登录失效视为正常故障。
3. 永远保留 FIT/CSV 手动导入，保证第三方接口变化时仍能补入活动数据。

旧 Garth 已停止维护且新登录失效，不能继续作为长期唯一方案。已有 Session 只可
作为迁移期输入，不把它写进代码、日志或 GitHub。

## 提醒与后台任务

- 第一版提醒是页内状态：未反馈、待同步、建议待确认和集成失效。
- iOS/iPadOS 16.4 起，加入主屏的 Web App 可以在用户主动点击授权后接收 Web
  Push；它是可选增强，不是应用可靠性的基础。
- Vercel Hobby Cron 最多每日执行一次且时间可能在目标小时内漂移，因此适合每日
  Garmin 同步和弱提醒，不适合分钟级调度。
- 自动任务必须持有锁或幂等键，重复触发不能重复写入。

## 实施顺序

### P0：交互与数据基础

- 共享 Layout、TanStack Query、聚合 API、客户端日期状态。
- 今日/日历缓存和底部 Tab 即时反馈。
- 组件级慢网络与快速切换回归测试。

### P1：当前康复闭环

- 出差/突发执行上下文、暂停和返程接续。
- 计划外训练、完整每日反馈、同步状态。
- 当前私有计划新版本导入与真机验证。

### P2：离线可靠性

- Dexie 事件队列、查询快照、启动/联网/聚焦重放。
- 冲突、重试、清除本机数据和 PWA 更新流程。

### P3：外部数据和镜像

- GitHub outbox 与数据仓库 Schema 快照。
- Garmin 适配器部署验证、手动导入、活动建议关联。

### P4：AI 和洞察

- DeepSeek 结构化建议、差异确认、版本应用与回滚。
- 趋势页、阶段评估、结项分流和周总结。

### P5：可选体验增强

- Web Push/角标、更多自动摘要、第二个 Tracker 模块。
