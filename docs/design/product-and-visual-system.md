# 产品与 UI/UX 设计系统

## 文档职责

本文是信息架构、页面结构、交互、视觉语言、组件状态和可访问性要求的权威设计规范。
产品目标与用户旅程见[产品需求](../product-requirements.md)，实施优先级和验收阶段见
[项目计划](../project-plan.md)。

## 设计方向

AK Tracker 的界面气质定义为：**安静、可信、有个人温度的康复日志**。

它不是医院信息系统，不使用冰冷、密集的表格来制造专业感；也不是竞技健身应用，
不使用高饱和色、大量速度感和挑战文案；更不是通用 SaaS 仪表盘，不把同步、配置
和技术指标放到使用者每天首先看到的位置。

视觉上使用暖白背景、白色或浅绿色表面、深森林绿主色和清晰的系统字体。交互上
接近原生移动应用的直接、稳定和可预测，但不模仿某一代 iOS 的装饰风格。我们希望
使用者打开应用时感到的是“我知道今天该做什么，而且记录不会丢”，而不是被数据、
警告或动画包围。

### 视觉语言速览

- **基底**：暖灰绿色页面承接长时间阅读，白色表面承载任务、反馈和详情，浅绿色
  表面表达选中、分组与完成后的低权重状态。
- **品牌感**：深森林绿用于主操作、当前导航和关键数字；品牌识别来自稳定的颜色、
  字体层级和语气，不依赖装饰性背景。
- **形状**：卡片以 `16px` 圆角形成清晰分组，按钮和输入使用 `12px` 圆角，状态
  标签使用全圆角；同一层级只使用一种主要圆角尺度。
- **层次**：正文卡片以细边框区分页面，悬浮导航、Sheet 和对话框才使用轻阴影；
  状态变化主要依靠表面颜色、描边、图标与文字共同表达。
- **图标**：使用圆润、简洁的线性图标，常规尺寸 `20–24px`、统一笔画粗细；当前
  Tab 可使用实心或更高对比版本，图标始终配合文字标签。
- **信息密度**：首屏保留当天结论、第一项任务和反馈状态；详细处方、实际记录、
  数据来源和技术状态按需展开。
- **图形表达**：趋势图优先展示可解释的变化和时间范围；恢复进度使用克制的曲线、
  区间和阶段标记，保持健康日志的稳定感。

## 产品参考与融合结果

参考产品用于校准界面结构、信息层级和交互质量，不作为逐像素模板。AK Tracker 的
最终视觉规范仍以本文件中的页面、组件和设计 Token 为准。

| 参考产品                                                                                         | 重点参考                         | 在 AK Tracker 中的落地                                                 |
| ------------------------------------------------------------------------------------------------ | -------------------------------- | ---------------------------------------------------------------------- |
| [Guava Health](https://guavahealth.com/article/guava-ultimate-guide)                             | 健康日志的浅色界面、卡片与时间线 | 作为整体视觉骨架：安静表面、模块化健康记录、明确的数据时间与来源       |
| [PhysiApp](https://www.physiapp.com/)                                                            | 康复任务、动作处方和完成反馈     | 训练以可勾选任务卡呈现，计划内容、实际完成情况和疼痛反馈彼此关联       |
| [Visible](https://www.makevisible.com/blog/introducing-updated-today-page)                       | 今日状态与低认知负担             | 首屏先给当天最重要的行动与安全状态，让用户一次扫视即可决定下一步       |
| [Gentler Streak](https://gentlerstories.com/gentlerstreak)                                       | 温和语气、阶段与进度表达         | 计划调整和康复总结使用非评判性文案，强调可持续进展和当前身体条件       |
| [Bearable](https://bearable.app/health-tracker/)                                                 | 可组合的日常反馈与多次记录       | 反馈由稳定问题、条件展开项和主观补充组成，允许一天内追加并保留时间顺序 |
| [Oura on the Web](https://support.ouraring.com/hc/en-us/articles/43392924561171-Oura-on-the-Web) | 清晰的健康趋势层级               | 趋势先给文字摘要，再给时间范围、图表、单位和来源，避免直接堆放原始数字 |

融合后的界面模型是：以 Guava 的浅色健康日志结构承载页面，以 PhysiApp 的康复任务
交互完成每日执行，以 Visible 的信息优先级组织今日页，以 Gentler Streak 的语气表达
进展与调整，以 Bearable 的模块化方式记录反馈，并以 Oura 的层级呈现长期趋势。不同
来源的做法通过同一套色彩、排版、间距和状态语义统一起来。

## 设计原则

1. **今日优先**：打开后先回答今天做什么、是否已反馈、是否有异常。
2. **交互先于网络**：点击必须先产生视觉和状态反馈，数据随后更新。
3. **安全信息分层**：普通计划、需要留意和需要停止不能争夺同一视觉权重。
4. **少而稳定的导航**：四个顶层 Tab 固定存在，页面内部再使用卡片、分段控件和
   模态流程。
5. **数据状态诚实**：本地保存、云端同步、GitHub 镜像和外部数据来源分别显示，
   不用一个“成功”混在一起。
6. **隐私默认**：锁屏通知和列表摘要不展示具体疼痛、诊断或训练备注。

## 从 Apple 设计原则选择性吸收

参考的 `apple-design` Skill 本身是把 Apple 的流畅界面思想翻译到 Web。AK Tracker
只采用适合手机 PWA 和康复场景的部分，不把它当成完整视觉模板。

| 原则         | 在本项目中的使用                                                      | 明确限制                                                                          |
| ------------ | --------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| 即时响应     | 按下按钮、日期和任务选框时立即出现按压或选中反馈；数据随后同步        | `pointerdown` 只负责视觉反馈，业务提交仍在 `click` / 触摸抬起时发生，允许取消误触 |
| 用户控制     | 任务只由用户确认完成；AI 和 Garmin 只建议；计划变更可预览、拒绝和回滚 | 不用自动动画或倒计时替用户确认                                                    |
| 空间一致性   | Sheet、弹层和详情从触发它的位置或方向出现，并沿相同路径返回           | 顶层页面不使用无意义的左右飞入动画                                                |
| 可中断       | 导航、日期切换和轻量弹层在动画期间仍能接受新输入；最后一次操作获胜    | 不让过渡动画锁住控件，也不等待动画结束才请求数据                                  |
| 简洁而非极简 | 常用路径直接可见，高级处方、同步原因和历史差异放到下一层              | 不为追求空白而隐藏今日反馈、风险状态或数据来源                                    |
| 清晰反馈     | 状态、完成、警告、错误使用不同强度，并尽量放在所影响内容附近          | 普通同步状态不弹全屏提示；真正红灯才提高打断等级                                  |
| 系统排版     | 使用系统字体、相对字号、随文字大小伸展的布局                          | 不加载只为装饰的字体，不固定所有字距和行高                                        |
| 减少动态效果 | 支持 `prefers-reduced-motion`；位移动画可退化为淡入淡出或无动画       | 不把动画作为理解状态的唯一方式                                                    |
| 半透明材质   | 最多用于固定底部导航或轻量 Sheet，且必须有高对比实色回退              | 不叠加玻璃卡片，不使用动画模糊，不让正文漂浮在复杂背景上                          |
| 弹簧与手势   | 只在以后确有可拖动 Sheet 等真实物理交互时使用临界阻尼、无弹跳弹簧     | 不为普通按钮、Tab、日期和卡片引入 Motion 依赖                                     |

不采用甩动、惯性投掷、橡皮筋边界、手势专属关键操作、网页声音或触觉反馈。它们
对当前以阅读、勾选和表单为主的产品帮助有限，却会增加误操作、性能和无障碍成本。

## 顶层导航

底部固定四个单词标签：

| Tab  | 核心问题             | 主要内容                                     |
| ---- | -------------------- | -------------------------------------------- |
| 今日 | 我今天要做什么？     | 任务、反馈、异常、待确认事项                 |
| 日历 | 某一天发生了什么？   | 月摘要、未来计划、历史详情                   |
| 趋势 | 康复是否在进展？     | 周完成、症状、负荷、睡眠背景、版本时间线     |
| 设置 | 数据和集成是否正常？ | 计划、同步、Garmin、AI、镜像、通知和本机数据 |

Tab 只用于导航，不放“新增反馈”“同步”等动作。不能根据页面状态隐藏 Tab，也不
长期展示不可点击的假 Tab；尚未完成的页面应提供明确空状态。Apple 的设计指南
同样建议 Tab 保持稳定、保留各自导航状态，并避免隐藏或禁用不可用项目。

## 页面框架

### 今日页

```text
日期 / 当前阶段                         刷新
执行模式 · 在线/离线 · 同步摘要

安全或中断横幅（仅需要时）

今日计划
  任务卡 1
  任务卡 2

身体反馈
  今日是否已反馈 / 追加反馈

待处理
  Garmin 关联 / AI 建议 / 同步错误
```

任务勾选、实际训练和反馈是最高频动作，保持在首屏可达范围。处方详情、历史差异和
技术性同步信息使用折叠或二级页面，避免挤占执行区域。

移动端首屏优先达到以下信息顺序：

1. 今天的日期、阶段和执行模式。
2. 是否存在需要停止或调整的安全信息。
3. 今天还剩多少任务以及第一项任务。
4. 是否已经完成当天反馈。

刷新按钮是显式、可触达的辅助入口。已有缓存时刷新不能清空当前内容或冻结页面，
只在标题附近显示轻量进度。

### 日历页

- 月份切换、星期标题和日期格组成稳定卡片；日期格触控区域至少 44×44 CSS px。
- 数字显示完成/计划；反馈、异常和出差等使用形状或图标，不能只靠颜色。
- 日期点击立即高亮；详情区独立显示缓存、骨架、错误和重试。
- 月份切换保留当前日号的意图；不存在的日期回落到该月最后一天。

### 趋势页

- 顶部先给一句可解释摘要，例如“本周完成 3/4，较上周少 1 次”，再显示图表。
- 左右侧数据始终可区分，并提供图例、单位和原始记录入口。
- 安全事件用图标和文字标记；不要用平滑曲线掩盖离散反馈。
- 睡眠、Garmin 活动只标为背景数据，不给出因果结论。

### 设置页

按用户心智分组：

1. 当前 Tracker 与计划。
2. 数据同步与离线记录。
3. Garmin、DeepSeek、GitHub 镜像。
4. 提醒和通知。
5. 隐私、导出、清除本机数据和退出登录。

## 页面与组件规格

### App Shell

- 手机内容宽度占满视口，左右安全边距默认 `16px`；宽屏内容上限约 `720px`。
- 底部导航位于受保护共享 Layout，页面切换时不重建；包含图标和文字，不只放
  图标。
- 底部导航考虑 `safe-area-inset-bottom`。支持半透明时只使用一层克制材质；
  `prefers-reduced-transparency` 不可用或用户提高对比度时使用实色表面。
- 每个 Tab 保留自身滚动位置和临时表单状态；身份变化时统一清除。

### 状态头部

- 第一行是日期或页面标题与刷新入口；第二行显示阶段、执行模式和同步摘要。
- “已同步”使用低权重文字，不制造持续的绿色徽章噪音。
- “仅保存在本机”“需要处理”等异常状态提升为带图标的状态胶囊，并能进入详情。

### 任务卡

- 默认只显示完成选框、任务名、核心剂量摘要和展开入口。
- 点击任务正文展开处方和实际记录；点击选框只改变完成状态，不同时展开卡片。
- 实际重量、组次、距离和主观感受独立保存，不能因勾选完成而自动填充。
- 完成状态降低视觉重量但保持文字可读；跳过状态同时显示原因入口。
- 保存失败时保留输入并显示“仅保存在本机”或“需要重试”，不能让卡片恢复成
  仿佛从未填写。

### 身体反馈

- 今日页使用一张摘要卡说明“尚未反馈 / 已反馈 N 次”和最近一次安全级别。
- 完整反馈在手机上使用独立全屏流程，不塞进狭小 Sheet；桌面可使用宽模态页。
- 高频问题先出现，异常问题按条件展开，主观感受最后填写；允许保存草稿。
- 提交后停留在结果页，明确展示安全级别、下一步和“继续添加反馈”，不立即把
  使用者丢回首页。

### 日历

- 日期格本身是按钮，触控区域不小于 `44×44px`；按下立即产生低强度背景反馈，
  选择后再显示稳定描边或填色。
- 月摘要只显示完成/计划数字和少量形状标记。反馈、出差、暂停和红黄灯不能全部
  变成不同颜色的小圆点。
- 月历保持稳定，日详情独立加载；切换日期不让整个日历闪烁或回到顶部。
- 第一版保留明确的上月、下月按钮，不依赖左右滑动。只有完成可用性验证后才考虑
  增加与按钮等价的可选手势。

### Sheet、对话框与独立页面

- Sheet 只承载轻量、可快速完成的选择，例如当天执行条件或同步错误详情。
- 长表单、AI 计划差异、返程接续和阶段评估使用独立页面，获得稳定地址、返回和
  草稿恢复能力。
- 只有不可逆删除、丢失未同步数据等真正高风险操作使用确认对话框；普通完成和
  保存依靠撤销或状态反馈。
- 如果以后加入可拖动 Sheet，手指拖动期间保持 1:1 跟随并允许中途反向；没有这类
  真实手势时不引入弹簧动画库。

### AI 建议与差异

- 顶部先显示“为什么建议调整”和安全级别，再按日期显示旧任务与新任务。
- 接受、拒绝、稍后处理三个结果明确；接受是主操作，拒绝不是危险操作。
- 红灯、过期和基础版本冲突时移除“接受”能力，并用文字说明下一步，不只禁用
  一个没有解释的按钮。

### 趋势图

- 图表之前先显示一句可解释结论和统计范围。
- 触摸数据点后在同一区域显示日期、单位和数据来源；不依赖悬停。
- 左右侧、计划完成、Garmin 和睡眠使用标签、线型或形状共同区分，颜色只是辅助。

## 关键流程

### 记录训练

- 任务卡的完成勾选与“实际完成情况”分开；勾选不自动编造组次、重量或距离。
- 保存后显示“本机已记录 / 正在同步 / 已同步 / 需要处理”。
- 计划外训练使用同一实际训练 Schema，但来源标记为计划外。

### 提交反馈

- 高频字段优先，异常问题逐步展开，主观感受最后填写。
- 首次保存不关闭整个上下文；页面显示安全结果和后续行动。
- 突发反馈是追加记录，不覆盖晨间或训练后反馈。
- 表单在页面内即时校验；提交失败保留全部输入和本机草稿。

### 计划调整

- 建议页使用逐项 diff：原任务、建议任务、原因、影响日期、安全级别。
- 接受和拒绝按钮视觉权重清楚；红灯或过期建议不提供直接应用。
- 接受前必须显示会生成的新版本和可回滚说明。

### 出差与暂停

- 今日页顶部显示执行模式横幅，不把所有备选训练同时塞进普通任务列表。
- 用户先描述今天的条件，再展示一个最合适的私有方案；仍可切换其他方案。
- 返程时使用独立接续向导，回顾实际完成与反应后生成未来计划预览。
- 当天执行条件可用轻量 Sheet 选择，但具体备选训练仍在任务卡中清晰展示。

### 阶段评估

- 评估分成多个短步骤，可保存草稿，左右侧分别记录。
- 最后展示达标项、未达标项和下一阶段选项；日历到期本身不代表成功或失败。

## 色彩与语义

沿用克制的绿灰体系。以下是第一版设计 Token 起始值，原型和无障碍测试可以微调
亮度，但不能改变语义角色：

| Token          | 起始颜色              | 用途                     |
| -------------- | --------------------- | ------------------------ |
| `page`         | `#F4F6F2`             | 页面暖灰绿背景           |
| `surface`      | `#FFFFFF`             | 主卡片和输入表面         |
| `surface-soft` | `#EEF3EF`             | 选中、分组和低权重状态   |
| `ink`          | `#18211E`             | 正文和主要标题           |
| `muted`        | `#5F6964`             | 次要说明、时间和来源     |
| `border`       | `#DCE3DE`             | 卡片、输入和分隔         |
| `brand`        | `#173F35`             | 主操作、选中状态、品牌   |
| `brand-soft`   | `#DDEBE5`             | 品牌浅表面               |
| `attention`    | `#994B19` / `#FFF3E8` | 待反馈、待同步、普通提醒 |
| `success`      | `#246B52` / `#E7F4ED` | 已完成、绿灯、同步成功   |
| `warning`      | `#7A5200` / `#FFF4CC` | 黄灯、需要调整           |
| `danger`       | `#A52820` / `#FDECEA` | 红灯、停止、不可应用     |

所有安全和同步状态同时提供文字与图标：`✓ 绿灯`、`! 黄灯`、`× 红灯`。不得只用
红绿颜色表达状态。普通“待处理”使用橙色，不滥用红色。

普通卡片默认使用边框而非阴影。只有悬浮导航、Sheet 和对话框使用低强度阴影；
不使用大面积渐变、霓虹、纹理背景或多层透明卡片。

## 界面元素视觉规范

| 元素         | 视觉规格                                                                | 状态与使用规则                                                       |
| ------------ | ----------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 主按钮       | 深森林绿实色、白色文字、`12px` 圆角、高度不小于 `44px`                  | 每个操作区域只保留一个主按钮；按下立即出现轻微颜色或缩放反馈         |
| 次按钮       | `surface-soft` 背景、深绿色文字、`1px` 可选边框                         | 用于保存草稿、稍后处理、查看详情等不改变主要路径的动作               |
| 文字按钮     | 透明背景、深绿色文字，触控区域仍不小于 `44×44px`                        | 用于刷新、展开、返回和低频辅助动作                                   |
| 卡片         | 白色背景、`1px border`、`16px` 圆角、默认无阴影                         | 任务、反馈和日详情保持同一表面层级；完成后转为浅绿色表面             |
| 输入控件     | 白色或浅色背景、清晰标签、`1px border`、`12px` 圆角、高度不小于 `44px`  | 聚焦时提高描边对比；错误文字紧邻字段并保留用户已经输入的内容         |
| 完成选框     | `24px` 左右的圆角方框，未完成使用中性描边，完成使用品牌色填充和勾选图标 | 整行标签扩大触控区域；任务勾选与实际训练数据分别保存                 |
| 状态标签     | 浅色语义表面、同色深文字、图标与短文案，全圆角                          | 用于绿黄红灯、待反馈、仅本机保存等状态；不单独依赖颜色               |
| 底部导航     | 四等分固定布局，页面色或实色浅表面，上方细分隔线                        | 当前项的图标和文字使用品牌色；每项同时显示图标与文字并保留导航状态   |
| 日期格       | 至少 `44×44px`，数字居中，任务数字和事件标记位于稳定位置                | 按下立即反馈，选中使用品牌浅表面或稳定描边，今天与选中日期可同时识别 |
| 趋势图       | 白色绘图区、清楚坐标与单位，品牌绿为主序列，语义色只标记安全事件        | 触摸后在图内展示日期、值和来源；图表上方始终提供同范围的文字摘要     |
| 空状态与骨架 | 复用最终卡片尺寸和页面结构，使用短说明与明确下一步                      | 首次加载显示局部骨架；无计划、无反馈和无数据分别使用对应的真实说明   |

## 排版、间距和触控

- 字体栈以 `system-ui`、`-apple-system` 和系统中文字体为主，不加载自定义字体。
- 页面日期/核心数字约 `28px/34px`、页面标题 `22px/28px`、区块标题
  `18px/24px`、正文 `16px/24px`、标签 `15px/20px`、辅助文字不小于
  `13px/18px`。
- 标题使用字号、字重和行高共同建立层级；大标题可轻微收紧字距，正文保持正常
  字距。尺寸使用 `rem`，布局需承受系统较大字体和 200% 文本缩放。
- 采用 4 px 基础间距；常用间距为 8、12、16、24、32 px。
- 卡片圆角默认 `16px`，输入与按钮 `12px`，状态胶囊使用全圆角；圆角服务于分组，
  不把每一行都做成独立胶囊。
- 主要触控目标至少 44×44 CSS px，日期格、Tab、复选框标签都扩大整个可点击区。
- 底部操作考虑 `env(safe-area-inset-bottom)`；主内容不能被 Tab 遮挡。
- 桌面仍以手机宽度为主，宽屏可把列表与详情分为两栏，但不改变信息顺序。

## 文案与信息表达

- 使用“已同步”“仅保存在本机”“正在同步”“需要处理”等用户语言，不显示
  outbox、Schema、HTTP、provider 等内部术语。
- 按钮使用具体动作：“填写反馈”“保存训练记录”“查看调整内容”，避免单独使用
  “确定”“提交”“处理”。
- 安全提示说明事实、动作和时机，不做医学诊断，也不使用夸张或责备语气。
- 空状态回答“为什么这里为空”和“接下来可以做什么”，不只显示插画或“暂无数据”。
- 时间和来源靠近对应数据，例如“今天 08:20 · 手动记录”“昨天 · Garmin”。

## 加载、错误与同步状态

- **首次且无缓存**：显示与最终布局等高的局部骨架。
- **有缓存并刷新**：继续展示缓存，仅在标题旁显示轻量刷新状态。
- **离线**：显示缓存时间和待同步数量；不遮挡任务。
- **写入失败**：记录仍保留在卡片中，提供重试和查看原因。
- **外部集成失败**：只影响对应卡片，Garmin、AI、GitHub 不能占据全屏错误页。
- 状态变化使用 `role="status"` 或等价可访问语义，避免只用短暂 Toast。

## 动效

- 可按压控件在按下时立即改变背景、描边或缩放；缩放只用于独立按钮，幅度约
  `0.98`，时长约 `80–120ms`。业务动作仍在触摸抬起后提交。
- 日期选中、任务完成和 Tab 状态使用 `120–180ms` 的轻量颜色、透明度或小幅
  transform 过渡，不做整页滑动。
- 所有动画必须可被下一次输入打断；快速切换日期时不排队播放旧动画。
- 网络内容加载不冻结控件，不使用全屏 Spinner。
- 支持 `prefers-reduced-motion`，关闭缩放、位移和装饰性动画，保留即时颜色与文字
  状态反馈。
- 半透明表面必须有实色回退，并尊重可用的减少透明度或提高对比度偏好。
- 动效只解释状态变化，不掩盖服务器等待时间。

## 响应式与 PWA 细节

- 重点验证 `320px`、`375px`、`390px` 和 `430px` 手机宽度；较窄屏允许标题换行，
  不缩小触控区域。
- 输入键盘弹出时，当前字段、错误和保存入口仍可见；不让固定底栏遮挡输入。
- iPhone 主屏模式没有浏览器刷新按钮，因此应用标题区持续提供刷新入口；发现新
  Service Worker 时给出“有新版本，刷新更新”的非阻塞提示。
- 桌面日历和趋势可使用列表/详情双栏；今日页保持单一主任务流，不扩展成后台
  仪表盘。
- 首次无缓存只显示局部骨架；有缓存时继续显示内容并后台刷新，避免页面跳回空白。

## 通知

- 第一版在页面内提醒，不要求通知权限。
- Web Push 必须由设置页中的明确按钮触发授权；拒绝后不反复弹窗。
- 锁屏文案只使用“今天还有一项记录待完成”等抽象描述。
- App Badge 只用于真正需要处理的数量，完成后及时清零。

## 可访问性验收

- 文字与交互符合 WCAG 2.2 AA；关键触控目标尽量达到 44×44。
- 不使用颜色作为唯一信息；图表有文本摘要、单位和可读表格替代。
- 键盘焦点可见，表单错误与字段关联。
- 异步保存、加载和错误状态能被辅助技术感知。
- iPhone 主屏模式、Safari、减少动态效果和较大字体均需真机验证。

## UI/UX 原型与验收流程

1. 先制作手机尺寸的“今日”和“日历”交互原型，覆盖正常、加载、离线、待同步和
   红黄灯状态。
2. 在原型中实际完成一次任务记录、一次身体反馈和一次日期快速切换，不只查看
   静态截图。
3. 再补出差执行条件、AI 计划差异、趋势和设置；复用已经验证的组件与状态语言。
4. 实现后以[核心场景测试](../testing/core-scenarios.md)中的慢网、离线、较大字体、
   退出清理和 iPhone 真机用例验收。

UI 实现不要求逐像素复制 Apple 平台，也不为了视觉相似而引入原生应用专属模式。
验收标准是：手机上快速、清楚、可预测、可撤销，并且安全信息与私人数据得到正确
对待。
